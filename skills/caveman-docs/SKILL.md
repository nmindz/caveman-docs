# Caveman-Docs: AI Documentation Compression

Active when writing AI-oriented documentation. **Human docs exempt.**

## Scope

**Apply caveman-docs compression to:**
- `AGENTS.md`, `CLAUDE.md` in any repo
- `docs/agents/*.md` (project_overview, architecture, tech_stack, coding_guidelines, domain_rules, api_contracts, data_model, dependencies)
- `SKILL.md` files in any plugin/skill directory
- `agents/*.md`, `commands/*.md` in agentic harness repos
- `.claude/*.md`, `.cursor/rules/*.md`, `.windsurf/rules/*.md`
- Any file explicitly described as "AI agent documentation" or "AI context"

**Exempt (write normally, full prose):**
- Confluence pages (via MCP or manual)
- Jira issue descriptions, comments, acceptance criteria
- `README.md` files intended for human developers
- PR descriptions and commit messages
- `CHANGELOG.md`
- Any doc the user says is for human readers

## Rules

### 1. No meta-prose
Heading states the topic. Content delivers facts. Never write a sentence that describes what the section is about.

Bad: "This section provides an overview of the build process for this project."
Good: (heading says "Build" — start with the command)

### 2. Bullets > paragraphs
Enumerable facts → bullet list. Prose paragraphs only for non-list narrative (rare).

Bad:
> The service uses PostgreSQL for persistence. It connects via PgBouncer. Migrations run with Flyway.

Good:
> - Persistence: PostgreSQL
> - Connection pool: PgBouncer
> - Migrations: Flyway

### 3. Tables > lists (when data is tabular)
Name+description, file+purpose, tool+version → table.

Bad:
> - `npm test` — runs all tests
> - `npm run lint` — runs ESLint
> - `npm run build` — compiles TypeScript

Good:
> | Command | Purpose |
> |---------|---------|
> | `npm test` | all tests |
> | `npm run lint` | ESLint |
> | `npm run build` | compile TypeScript |

### 4. Code blocks verbatim
Commands, file paths, code examples — unchanged, exact. Never paraphrase a command.

### 5. Drop hedging
"typically", "usually", "generally", "in most cases", "it is worth noting", "please be aware", "note that" → delete.
If conditional, state the condition explicitly. If always true, state it.

### 6. Active voice, imperative
"Run tests with `npm test`" not "Tests can be run using `npm test`."
"Never import across layer boundaries" not "It is generally advised not to import across layer boundaries."

### 7. One fact per bullet
Multi-sentence bullets only when the second sentence is a code example or explicit constraint.

### 8. No disclaimers or ownership commentary in body
"This file is generated by..." belongs in an HTML comment, not prose. Never in body bullets.

### 9. Numbers as digits
"3 patterns" not "three patterns". "120 bytes" not "one hundred and twenty bytes".

### 10. Abbreviations OK
DB, API, CLI, auth, config, repo, impl, deps, env — use freely. Spell out only on first use if genuinely ambiguous.

## Compression Examples

### AGENTS.md build section

Bad (verbose):
```
## 1. Build, Lint, Test and Coverage

Before making any changes to this project, you should familiarize yourself
with the available build and test commands. It is generally recommended
to run the linter before submitting a pull request.

To run the tests, use the following command:
```bash
npm test
```

You can also check code coverage with:
```bash
npm run test:coverage
```
```

Good (caveman-docs):
```
## 1. Build, Lint, Test and Coverage

| Command | Purpose |
|---------|---------|
| `npm test` | all tests |
| `npm run lint` | ESLint |
| `npm run build` | compile TypeScript |
| `npm run test:coverage` | coverage report |

Coverage target: ≥ 80%. CI blocks on failure.
```

### docs/agents/architecture.md

Bad (verbose):
```
# Architecture

This document provides an overview of the architectural patterns used in
this project. Understanding the architecture is important before making
any changes to the codebase.

## AS IS — Current state

The project follows a hexagonal architecture pattern, which is also known
as ports and adapters. This means that the domain logic is isolated from
external concerns such as databases and HTTP frameworks.
```

Good (caveman-docs):
```
# Architecture

## AS IS — Current state

### Style
Hexagonal (ports and adapters). Domain isolated from infrastructure.

### Package layout
```
src/
├── domain/      # entities, value objects, domain services
├── application/ # use cases, ports (interfaces)
├── adapters/    # HTTP controllers, DB repositories, queue consumers
└── infra/       # framework config, DI wiring, migrations
```
```

### SKILL.md frontmatter description

Bad: "This skill provides guidance for working with the authentication module in this repository, helping AI agents understand the patterns and conventions used."

Good: "Auth module patterns — JWT flow, token refresh, permission guards. Use when modifying auth/* files."

## Intensity Levels

| Level | Effect | Scope |
|-------|--------|-------|
| `lite` | Drop filler/hedging/pleasantries. Keep articles + full sentences | AI docs |
| `full` (default) | Drop articles, fragments OK, short synonyms | AI docs |
| `ultra` | Max compression. Abbreviate (DB/auth/config/req/res/fn/impl). Strip conjunctions. Arrows for causality (X → Y) | AI docs |
| `ultimate` | Same as `ultra` PLUS intercept ANY `.md` write (any path/type) | All `.md` (except exempt list) |
| `wenyan-lite` | Semi-classical Chinese. Grammar intact, filler gone | AI docs |
| `wenyan` / `wenyan-full` | Full 文言文. Max classical terseness. 80-90% character reduction. Particles 之/乃/為/其 | AI docs |
| `wenyan-ultra` | Extreme classical compression. Ancient scholar on a budget | AI docs |
| `wenyan-ultimate` | Same as `wenyan-ultra` PLUS intercept ANY `.md` write | All `.md` (except exempt list) |
| `off` | Disable | — |

### Mode resolution order
1. Flag file `~/.claude/.caveman-docs-active` (set by `/caveman-docs <level>`)
2. `CAVEMAN_DOCS_DEFAULT_MODE` env var
3. `CAVEMAN_DEFAULT_MODE` env var (shared with upstream `caveman` plugin)
4. `~/.config/caveman-docs/config.json` `defaultMode` field
5. `full`

### Switch mid-session
- `/caveman-docs lite|full|ultra|ultimate|wenyan-lite|wenyan|wenyan-ultra|wenyan-ultimate|off`
- Bare `/caveman-docs` → re-resolve from env/config

### ULTIMATE behavior (`ultimate`, `wenyan-ultimate`)
- PreToolUse Write/Edit/MultiEdit fires on every `.md` (and `.markdown`) write — not only AI doc paths
- Exempt: `README.md`, `CHANGELOG.md`, `CONTRIBUTING.md`, Confluence, Jira, PR descriptions, commit messages
- Non-`.md` files (`.ts`, `.js`, `.py`, etc.) untouched

### Compression by level — same input, different output
Input: "Component re-renders because a new object reference is created on each render."

| Level | Output |
|-------|--------|
| `lite` | "Component re-renders due to new object reference each render." |
| `full` | "New object ref each render → re-render." |
| `ultra` | "Inline obj prop → new ref → re-render." |
| `wenyan-lite` | "組件頻重繪，以每繪新生對象參照故。" |
| `wenyan` | "物出新參照，致重繪。" |
| `wenyan-ultra` | "新參照→重繪。" |

## Auto-Clarity Exception

Write full prose when:
- The doc is explicitly marked as human-facing
- The user says "normal docs" or "human readable"
- Writing a Confluence or Jira artifact via MCP

Resume caveman-docs immediately after the exempt artifact is complete.

## Integration with /ai-context and /caveman-docs:init

When either command generates AI docs, compression is mandatory for every artifact except CLAUDE.md (already ≤400 bytes — just ensure the redirect is minimal). The `docs/agents/*.md` files must use tables and bullets throughout; prose paragraphs only in the "one paragraph justification" shape for N/A files.
